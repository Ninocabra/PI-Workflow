1. Introduction and General Principles of the Script

PI Workflow is a comprehensive astrophotography processing interface that unifies the entire post-processing pipeline into a single environment. It offers One Preview to Rule Them All: A single, high-performance, interactive preview panel shared across all tabs and tools. It also add new and powerful comparison toolslike Interactive Split-Screen (Split View), Multi-Algorithm Comparison Grid and 8-Slot of Transient Memory. Enjoy

Built around an architecture centered on canonical image ownership, the script behaves like a tutor that assists with the four universal processing phases: Pre Processing, Stretching, Post Processing, and Channel Combination. It does not replace PixInsight's native processes; it orchestrates them more efficiently by automatically applying metadata, controlling data linearity, and providing high-speed interactive previews combined with instant state memories.

Community Perspective: Workflow Consolidation

Reference astrophotographers such as Adam Block[1] and Cuiv the Lazy Geek[2] repeatedly point out that the main barrier in PixInsight is not the internal mathematics of its algorithms, but the difficulty of maintaining a structured workflow where linear relationships are strictly respected and "A/B" comparisons can be performed immediately. PI Workflow embodies that philosophy by isolating linear processing from non-linear processing and introducing a preview engine that preserves the spatial context of the image.

2. User Interface (UI) Architecture and Memory Management

The general UI design principle in PI Workflow divides the workspace into strict functional zones that interact dynamically with one another. This spatial arrangement allows the user to focus on aesthetic and physical decisions without losing sight of the raw data.

Main Visual Structure

  • Title Bar (Top): Displays the logo π, the active script version, and the strategic Repositories button. This button exposes a complete manifest of curated community repositories and external system dependencies.
  • Workflow Tabs (Top Level): Main navigation divided sequentially into the four master stages. When the tab changes, the script reconfigures the control panels and automatically updates the viewers so they point to the image layer corresponding to the current state.
  • Function Panel (Left): Contains execution controls, drop-down menus, and numeric controls organized in collapsible sections. At the top of this panel there is always a distinctive title for the active engine, for example PRE PROCESSING ENGINE, followed by the shared and persistent Image Selection block.
  • Integrated Preview Viewer (Right): The unified rendering canvas. It displays the active image, the candidates generated by a process, gradient model overlays, and real-time mask previews.

Shared Systems and Memory Management

One of PI Workflow's most critical technical innovations is its double transient-memory system coupled to a strict RAM consumption policy.

  1. Image Selection (Persistent): Present in all tabs, it allows the user to map master views open in the workspace to the script's internal keys (R, G, B, L, H, O, S, RGB, etc.). Quick mode buttons (R+G+B, NB, RGB) filter the interface according to the nature of the session.
  2. Global Memory Slots (Memory Slots 1-8): Located above the viewer on the right. They allow complete snapshots of either the candidate image or the current image to be captured with a single click.
    • Left click on a number: saves the active visual state, the signature of the algorithm used, and the internal metadata to that slot.
    • Right click on a number: loads the stored state back into the active canvas as a candidate key, allowing earlier processing branches to be recovered.
    • Reset button: immediately releases the transient memories of the active tab.
  3. Mask Memories (Mask Memory 1-8): Exclusive to the Post Processing and Channel Combination tabs. They store dynamic masks generated at full resolution for later local application or assignment to specific layers.
  4. Memory Budget (~1.5 GB): To prevent intensive storage of 32-bit floating-point images from exhausting system RAM and causing PixInsight crashes or unexpected shutdowns, the script implements a predictive garbage collector. If memory storage exceeds the soft limit configured in OPT_MEMORY_BUDGET_BYTES (1.5 GB by default), the script silently releases the oldest unselected slots until a safe operating margin is restored.

Zoom, Framing, and Resolution Control

The viewer on the right operates on a native ScrollBox container that overcomes the limitations of PixInsight's conventional previews:

  • Drag (Pan): By holding the left mouse button, the cursor changes to a closed hand (StdCursor_ClosedHand), allowing the frame to be moved freely.
  • Mouse Wheel (Live Zoom): Enlarges or reduces rendering scale from 5% up to 4000%, recalculating spatial position in real time so the pixel under the cursor remains strictly anchored.
  • Preview Resolution Reduction Factor: A configurable drop-down menu from 1 to 6. A value of 3, the default, means that for interactive calculations such as blur-slider adjustments or color-wheel feedback, the image is internally downsampled by dividing each side by 3. This dramatically speeds up graphic response. When the action is confirmed with Use this Image, the script always recalculates the algorithm on the full-resolution image.
  • Split-Screen Comparison (Split View): Toggled via the Split button next to the Zoom card. When active, the viewer splits vertically: the left side displays the original or previous image state (or previous memory slot), and the right side shows the active candidate or current image. Users can hover over the vertical amber separator line to see the horizontal resize cursor (StdCursor_SizeHor) and drag it to adjust the division ratio dynamically. Normal panning is temporarily locked while dragging the split slider.
  • Global Export Controls: The Export and Export TIF buttons have been unified at the top-right corner of the main window, directly next to the sequential workflow tabs. Clicking either button exports the active view of whichever processing tab is currently open, eliminating local duplicate buttons.
Best-Practice Guide: Strategic Use of Memory

Use memories 1 through 3 to compare parameter variations of the same process, for example different deconvolution strengths or background-extraction settings. Store one candidate in memory 1, change the parameters, regenerate the candidate, and store it in memory 2. Then toggle between them with right click to evaluate subtle structural differences before confirming with Use this Image.

3. Workflow Rationale and Philosophy

Astrophotography workflow is not arbitrary. It obeys immutable physical laws that govern photon collection and the additive or multiplicative nature of noise and signal. The PI Workflow script enforces a strict architectural separation between the Linear Domain and the Non-Linear Domain, stopping execution and throwing explicit errors if the user attempts to cross those boundaries incorrectly.

The implemented sequential logic follows this physical reasoning:

  1. Pure Linear Phase (Pre Processing): Freshly stacked data still maintains a direct linear relationship with the amount of light emitted by celestial objects. In this domain it is imperative to correct global physical anomalies:
    • Gradients: Light pollution and moon glow add parasitic light in an additive way, gradually distorting the background. Optical vignetting reduces signal toward the edges in a multiplicative way. These defects must be subtracted or divided out while the data is still linear.
    • Color Calibration: Captured radiometric fluxes must be referenced against real stellar catalogs such as Gaia DR3/SP to determine exact white balances. Doing this after stretching would completely invalidate the underlying color physics.
    • Deconvolution: Correction of point-spread light dispersion caused by atmospheric seeing and optics mathematically assumes linear image formation behavior.
  2. Structural Transition (Stretching & Star Split): Once the linear image is physically correct and sharp, we face a dynamic-range problem. Faint nebular structures barely occupy the first levels of the histogram, while stellar cores saturate quickly. If we stretch globally, stars will bloat disproportionately, blocking the background scene and losing their color profiles. That is why the current philosophy dictates performing a Star Split before non-linear stretching. The Starless layer is stretched hyperbolically to reveal faint gas, while the Stars layer receives a gentle logarithmic stretch to preserve hue and point intensity.
  3. Non-Linear Phase (Post Processing): With the structures already revealed and separated into layers, processing enters an aesthetic and perceptual phase. Here multiscale algorithms are applied to imitate and enhance human visual response: high dynamic range compression in bright cores, local histogram equalization to emphasize volumes of dust and gas, and luminance/chrominance noise reduction adapted to low-SNR areas.
  4. Recombination (Channel Combination): The final phase returns sovereignty to the user so previously isolated layers can be composed freely. It allows pure luminance injection over RGB bases, reintroduction of intact stellar profiles over heavily processed nebulae, or blending of pure emission channels such as Ha, OIII, and SII using advanced mathematical logic.

4. Tab 0: Pre Processing (Linear Domain)

The Pre Processing tab is the canonical entry point for images into the script. All operations in this section assume and verify that the selected view carries the property isLinear === true in its FITS/XISF header.

Strict Linearity Control

PI Workflow includes an internal safeguard directive in its background-extraction engines (MGC and native GraXpert) and color-calibration engine (SPCC). If an image that has already been stretched is loaded, the script aborts the operation immediately and displays the error: [PROCESS/LINEARITY] Process requires a linear image. The selected view appears to be non-linear. This prevents accidental corruption of the physical calibration.

4.1. Image Selection and RGB Assembly

In the left panel, the Image Selection block allows master files to be loaded. If the user has separate monochrome images (R, G, B, or narrowband lines H, O, S), they can be assigned individually in their drop-down menus and combined with the Combine RGB button in the Image Selection block. Internally, the script calls PixelMath to create a new canonical three-channel floating-point window, rigorously copying the astrometric metadata from the reference view, usually the Green or Ha channel, so spatial traceability remains intact.

Automatic image detection. Each drop-down in the Image Selection block is associated with a canonical slot name. When the script initialises or refreshes the selectors, it scans all open windows and pre-selects the one whose identifier matches the slot name exactly (case-insensitive). 4.2. Crop (Manual, Automatic, and Handle-Based Edge Removal)

Stacks often have defective borders (zero-value pixels) produced by stacking with dithering, rotation, or partial frame coverage. PI Workflow exposes a Crop section between Image Selection and Plate Solving that removes those defects with three coexisting interaction modes.

Interaction modes

  • Manual: Hold SHIFT and drag with the left mouse button on the preview to draw a rectangle. The unmodified left-button drag still pans the image, so the crop tool never interferes with normal navigation.
  • Automatic: Press Auto-detect Edges. The script scans each edge inward using PJSR's native image.minimum() on row/column sub-rectangles and identifies the largest interior region with no zero pixels. Cost is a few milliseconds even on 8K images thanks to a coarse-then-fine search (step 16 + 16-row refinement).
  • Handle editing: Eight handles (4 corners + 4 edge midpoints) appear on any active rectangle, regardless of how it was created. Drag a handle to resize that edge or corner. Drag the rectangle interior to translate it. Handle hit-test tolerance is 10 viewport pixels — handles feel the same size at every zoom level.

Applying the crop

  • Apply to Current — Crops the currently displayed image in place. The change uses PixInsight's native Crop process, so it is undoable from the image window's standard Undo history.
  • Apply to All — Applies the same rectangle to every loaded image of the active mode (R/G/B/L in MONO, H/O/S/L/HO/OS in NB, RGB in RGB). Because the crop offsets are identical, the images keep their relative alignment without any further action.
  • Checkbox Re-align after multi-crop — Optional. Runs StarAlignment after Apply to All using the first cropped image as reference. Produces new views suffixed _r; the original cropped views are left intact. Useful only if you suspect sub-pixel misalignment in the source frames — typical PixInsight integration output needs no re-alignment.

Astrometric metadata is preserved automatically. Copying the original FITS/XISF WCS headers verbatim to the cropped image would be incorrect because the reference pixel positions (CRPIX1, CRPIX2) point to specific pixels that have moved. The script uses the native Crop process, which shifts CRPIX1/2 by the crop offsets and updates NAXIS1/NAXIS2. The sky-referenced fields (CRVAL, the CD matrix, CTYPE) do not need updating — they describe the projection, not the pixel grid. Net result: plate-solved WCS survives cropping with no manual intervention.

4.3. Plate Solving (WCS Astrometric Solution)

Astrometric solving provides the image with absolute spatial coordinates through the World Coordinate System, or WCS. It is an unavoidable technical prerequisite so later processes can query astronomical databases.

  • Order and Action: Press the Solve Image button in the Plate Solving section.
  • Internal Logic: The script first evaluates whether the image already has a valid WCS solution by looking for projections or FITS keywords such as CTYPE1 and CRVAL1. If it does not, it silently configures a native instance of the central ImageSolver script.
  • Execution Parameters: Field-distortion correction is forced with distortionCorrection = true, and the advanced mathematical interpolation type is set to thin-plate splines with RBFType_DDMThinPlateSpline and an upper limit of 2000 anchor points to absorb severe geometric aberrations caused by focal reducers or misalignment.
  • Interaction and Fallback: If automatic solving fails due to missing focal-length or pixel-size metadata in the header, the script does not throw a useless error. Instead, it automatically opens the native interactive ImageSolverDialog so the user can manually enter approximate object coordinates, right ascension and declination, and retry the calculation. After success, the visual state is tagged with a green confirmation indicator.

4.4. Gradient Correction: MGC, Native GraXpert, AutoDBE, and ABE

Linear gradient correction removes unwanted spatial variations. The Algorithm drop-down menu offers four radically different technological approaches to this problem, allowing the user to compare their effectiveness by generating candidate previews.

One-click comparison across all algorithms

Next to Gradient Correction there is a secondary Compare button. Pressing it runs every available algorithm against the active image, stores each full-resolution result in its own Memory slot (Memory 1 = MGC, 2 = AutoDBE, 3 = ABE, 4 = GraXpert) and shows a labelled 2x2 mosaic of all variants in the preview pane so you can judge them side by side. Right-click any slot to inspect that variant at full resolution with the normal zoom and pan; then click Use this Image to commit the winner as the new working image. Reset Memory wipes all variants in one go. Algorithms that are not installed in the running PixInsight build show as empty mosaic tiles with a short reason; the variants of the installed algorithms are unaffected.

4.5. Color Calibration: SPCC, Auto Linear Fit, and Background Neutralization

Linear chromatic fidelity ensures that stars reflect their true surface temperatures and nebulae display the correct atomic transitions. In this block PI Workflow exposes three dedicated calibration actions, each designed for a different level of astrometric and photometric availability.

Automatic gating in monochrome workflows. When the canonical image of the Pre Processing tab is monochromatic (separate R, G, B or narrowband channels not yet combined), the entire Color Calibration block is automatically disabled and its controls show a tooltip explaining the reason. As soon as you press Combine RGB (or Combine HOS) in the Image Selection block, the section re-enables itself with no further action required. This is part of the centralized UI gating system (v33-opt-8k).

Use this when the background pedestal is tinted but the stellar chromatic slopes are already acceptable. Unlike SPCC or Linear Fit, it neutralizes only the background anchor and does not force a full photometric calibration of the highlights.

4.6. Linear Deconvolution: BlurXTerminator and Cosmic Clarity

Deconvolution reverses linear optical blur before stretching. The Algorithm drop-down menu offers two distinct restoration engines. PI Workflow keeps their native controls while wrapping them in a safe candidate-preview workflow so the user can compare sharpening strength before committing it to the master image.

One-click comparison across both engines

5. Tab 1: Stretching (Non-Linear Transition)

Crossing from the linear domain to the non-linear domain permanently alters the metric of the data. The main objective of stretching is to redistribute brightness values so structures of interest rise above the threshold of human visual perception without sacrificing global contrast.

5.1. Star Split (Star Separation and the Starless Philosophy)

At the top of the tab sits the Star Split section. It exposes an Algorithm drop-down that selects the engine used to separate the stars from the rest of the signal, the parameter panel of the selected engine, and the Split Stars call to action that runs it. Executing the split marks the central methodological milestone of the modern processing workflow.

Why Perform Star Split Before Stretching

If we apply an aggressive histogram-transform curve to an image that contains both stars and nebula together, the stellar pixels, which already carry high signal, will shoot toward saturation at 1.0. This causes physical widening of stellar disks, irrecoverable color loss in the cores, and the creation of a dense veil of stars that hides subtle background gas detail. By isolating stars during the linear phase, we obtain two pure canvases. The nebula can be subjected to extreme stretches, local equalization, and aggressive noise reduction without touching a single stellar pixel. The stars, meanwhile, can be stretched conservatively to preserve point profiles and pristine colors.

Live Preview, Promotion, and Mask Memory (v33-opt-9m unified flow)

When a mask Live checkbox is enabled, the main viewer renders the computed mask in real time. To keep interaction fluid, PI Workflow downsamples the preview so its longest side never exceeds 1024 pixels (OPT_POST_LIVE_MAX_DIM).

The mask system mirrors the image-memory model:

  • Live preview = staging. Adjust thresholds, hue, or FAME shapes; the preview updates in real time but the active mask is unchanged.
  • Use This Mask = commit. Builds the final full-resolution mask from the current parameters and promotes it as the active mask. From here, any Post-processing tool with Use active mask enabled (Noise Reduction, Sharpening, Color Balance, Curves) will respect it. This button is the mask-side equivalent of Use this Image in image memory.
  • Left-click on a mask memory slot 1–8 = store the current active mask there for later reuse.
  • Right-click on a mask memory slot = recall and activate that slot's mask in a single gesture — it becomes the new active mask immediately, no separate confirmation button needed. Same right-click semantics as the image-memory slots above.

Mask polarity — white means "the mask acts here". All three algorithms (Range Selection, Color Mask, FAME) follow the same rule: the white pixels of the mask are the pixels that receive the operation; the black pixels are protected. The script enables maskInverted = true on the underlying view before each Post process executes so this convention is enforced regardless of PixInsight's default. The overlay rendered on top of the image uses an amber-gold tint over the active (white) regions so it is visually consistent across designing, painting (FAME), and committed states.

Show/Hide Mask toggles the amber overlay on the preview without modifying the mask itself. Use it to compare masked and un-masked areas before applying a process.

7. Tab 3: Channel Combination (Composition and Multi-Layer Blending)

The Channel Combination tab is the final assembly engine. Its purpose is to give the user absolute control over the blending of master images or intermediate layers generated throughout the session, far surpassing the functionality of PixInsight's native process of the same name.

7.1. Blending Logic, Slot Assignment, and Per-Image Masks

The interface exposes a vertical stack of up to 6 independent slots. Each slot behaves like a layer in a non-linear editing application, but operates in PixInsight's strict floating-point space.

For each slot, the user configures:

  • Source: A drop-down menu listing all compatible views open in the workspace, as well as canonical images internally managed by the script, such as the previously stretched Starless and Stars layers.
  • Active: A checkbox that dynamically includes or excludes the layer from the global blending equation.
  • Live (Isolate Layer): A checkbox that forces the viewer on the right to render only the result of this individual slot, making fine tonal adjustment easier without visual interference from the rest of the stack.
  • Tone and Color Controls: Direct sliders for Brightness, which acts as multiplicative brightness, and Saturation. Enabling the Color checkbox reveals individual RGB multipliers and hue anchors. Enabling Histogram activates an exclusive Bezier-curves canvas to sculpt the tonal response of that single layer independently before it enters the global blend.

One of PI Workflow's most powerful capabilities is the ability to assign an exclusive local mask to each image slot through the Mask drop-down menu. Eligible masks are taken directly from views previously saved in Mask Memory.

Practical Use: Selective Layer Treatment

Suppose we are composing an image that blends a starless nebular layer with a stars layer. If we want to strongly saturate stellar colors without altering the background nebula at all, it is enough to assign the stars view to Slot 2, select a star mask in its Mask menu, and raise the Saturation control for that slot. Internally, the script applies the saturation increase weighted strictly by the local-mask values before adding the stellar layer to the global canvas.

Strict Geometry Validation: To guarantee mathematical precision, PI Workflow enforces a runtime verification. The mask selected in a slot must match the source image of that slot exactly in pixel dimensions. If the geometries differ, the script aborts the preview and throws an explicit error to avoid applying transformations to misaligned pixels.

7.2. Live Global Blending Without Loss of Scale (See all Images Blended)

PixInsight's native ChannelCombination process suffers from a classic and frustrating defect: when real-time preview is enabled, the native engine performs a double graphical resampling reduction that ends up drawing the composite result in the upper-left corner at a tiny size, destroying any previous framing or zoom level.

PI Workflow solves this problem at the root with a clean engineering solution in its See all Images Blended directive:

  1. When live global blending is enabled, the script intercepts the source views of all active slots.
  2. Instead of forcing PixelMath to compute the weighted sum of six floating-point 60-megapixel layers dozens of times per second, which would freeze the interface, the script pre-downsamples each source layer so it fits exactly inside a maximum bounding box of 1024 pixels defined by OPT_CC_LIVE_MAX_DIM, matching the actual on-screen resolution of the right viewer.
  3. It then compiles a multilayer PixelMath expression on the fly and executes it over these reduced ultra-lightweight versions in RAM.
  4. The composite result is injected directly into the native viewer, skipping the second reduction step of PixInsight's preview engine. The final effect is substantial: the user keeps the zoom level, pan framing, spatial position, and apparent resolution intact while watching the layers blend fluidly and instantly as opacity or brightness sliders are moved.

Canonical Consolidation: The fast downsampled preview is purely graphical. The moment the user makes the final aesthetic decision and presses Use this Image, the script invokes a candidate upgrader. This module discards the reduced caches, rebuilds the complete mathematical equation in PixelMath, and executes the composition layer by layer using the original full-resolution source images, depositing the resulting master canvas back into the central workflow store.

8. Advanced Narrowband-Aware Logic

Processing data captured through emission-line filters, that is, narrowband data, obeys calibration rules different from those used for broad-band light or traditional RGB. If we treat an SHO palette as though it were a daylight photograph, we destroy the physical information of ionized gases.

PI Workflow implements an intelligent Narrowband-Aware Framework embedded in the core of its linear and non-linear engines:

Palette Detection and Labeling

The script continuously monitors image origin. If the user assembles a file using individual channels explicitly assigned to emission lines, or selects predefined color recipes in the header such as SHO, HOO, HSO, HOS, OSS, OHH, OSH, OHS, HSS, REAL1, REAL2, or the popular dynamic FORAXX palette[17], the script automatically injects a set of standardized FITS keywords into the resulting image header:

  • PIWNB = 'T' for a script-managed narrowband RGB compound flag.
  • PIWNBREC = '<Recipe_Name>' for the palette identifier, for example SHO.
  • PIWNBR, PIWNBG, and PIWNBB for the explicit atomic mapping of each visual channel to its original emission line, H, O, or S.

Differential Behavior in Key Processes

The presence of these tags automatically modifies the internal execution of the PixInsight tools orchestrated by PI Workflow:

1. Spectrophotometric Color Calibration (SPCC)

When SPCC is executed on a compound tagged as narrowband, the script reads the emission lines assigned to each visual channel and automatically configures the native process parameters in Narrowband mode. It assigns the central wavelengths and characteristic bandwidths stored in its internal database OPT_NB_LINE_DB:

  • H-alpha (Ha): Wavelength = 656.28 nm · Default bandwidth = 7.0 nm.
  • OIII: Wavelength = 500.70 nm · Default bandwidth = 7.0 nm.
  • SII: Wavelength = 672.40 nm · Default bandwidth = 7.0 nm.

This ensures radiometric fluxes are compared correctly against Gaia stellar spectra filtered through those specific narrow passbands.

Physical Restriction: Rejection of Mono Channels in SPCC

PI Workflow intentionally forbids and blocks execution of SPCC on a view that represents a single pure monochrome channel, for example an individual Ha file. A single emission channel does not contain differential color information. Generating a pseudo-RGB copy of that channel just to force SPCC to run would yield mathematically absurd multipliers while ignoring filter physics. The correct methodology requires combining the channels into an RGB palette such as HOO or SHO first, and then applying SPCC to the resulting compound.

2. Prior Flux Calibration for Gradient Correction (SPFC / MGC)

For the physical gradient engine MGC to predict the background in a narrowband image, it requires the prior flux-calibration stage, SPFC, to receive exactly the same filter parameterization. PI Workflow injects these metadata on the fly into the SPFC instance before invoking MGC, ensuring that the satellite-based physical model understands which part of the luminous spectrum is being evaluated.

3. Protection of Emission Relationships During Stretching (MAS)

When the adaptive non-linear stretch MultiscaleAdaptiveStretch, or MAS, is applied to a traditional broad-band image, background equalization is desirable. However, if we apply strict background equalization to an SHO palette where the SII channel, red, is naturally much weaker than the Ha channel, green, the algorithm will aberrantly boost SII noise to force its histogram toward the hydrogen channel.

PI Workflow's framework detects this situation: if the image is tagged as a narrowband compound, the script automatically disables broad-band background equalization inside MAS. This preserves the real radiometric intensity relationships among ionized gases, allowing the coveted gold and turquoise transitions of the Hubble palette to emerge naturally and with physical coherence after the stretch.

10. Dependency Checks and Troubleshooting

To give the session maximum technical transparency, PI Workflow compiles and executes a complete Dependency Checks Registry in the background during initialization. This report classifies the operating state of each external module into three severity levels: OK, WARN for non-critical warnings that enable safe fallbacks, and ERROR for functional blocking of an algorithmic branch.

The user can consult the detailed system state by going to the script's final configuration page or by pressing the Refresh Dependency Check button.

Common Error Resolution Matrix

MGC Error: "SPFC Scale Factors not found"

Physical Cause: The MGC engine attempts to model physical light dispersion, but the image lacks the internal PCL:SPFC:ScaleFactors property in its FITS header.
Direct Solution: MGC cannot run blindly. Make sure plate solving has been completed and create a process icon named exactly SPFC, or SPFC_NB for narrowband, on the PixInsight desktop, configured with your sensor's QE curve. When gradient correction is retried, PI Workflow will intercept the error, silently run your SPFC icon on the image to inject the scale factors, and then complete the MGC execution successfully.

SPFC Error: "Gaia / XPSD database not found"

Physical Cause: The SPFC process has been invoked, but PixInsight does not have the local path to the Gaia DR3/SP spectrophotometry database files, with extension .xpsd, configured.
Direct Solution: Download the Gaia DR3/SP database files, either the small or full packages, from PixInsight's official distribution site. Then go to Processes > Photometry > SpectrophotometricColorCalibration, click the wrench icon in the Gaia database section, and add the folder where you stored the .xpsd files. Save the global configuration.

Denoise/Decon Notice: "Cosmic Clarity / ExternalProcess not available"

Technical Cause: You selected the Cosmic Clarity engine in the deconvolution or noise-reduction sections, but the script cannot locate the external SetiAstro executable, or the active PixInsight build restricts use of the PJSR ExternalProcess class because of operating-system security policies.
Direct Solution: Verify that you have purchased and installed the standalone SetiAstroSuitePro application. If the error persists, switch the algorithm drop-down to the fully integrated native alternatives: BlurXTerminator for deconvolution, and NoiseXTerminator or native GraXpert Denoise for noise attenuation.

Unexpected Blending Behavior: "The combine button does nothing"

Technical Cause: When trying to execute stretches with the legacy VeraLux suite or combine channels through older external scripts, the graphical interface may appear to ignore the click, throwing no error and not updating the viewer. This happens when the internal collector captures global object constructors, such as a Windows DLL exposing globals with the same process name, and incorrectly evaluates them as executable functions instead of instantiating the process class.
Direct Solution: PI Workflow addresses this by implementing a strict resolver, optResolveVeraLuxProcessFunction, which validates parameter signatures and prioritizes direct instances created from environment icons. If you experience extreme graphical slowdown during complex combinations in the final tab, enable See all Images Blended to force rendering through the unified high-speed downsampled engine.

11. Summary of Best Practices and Expert Tips

To extract the maximum analytical and aesthetic performance from PI Workflow, incorporate the following guidelines into your processing routine:

  1. Non-Negotiable Linear Cleanup:[1] Never try to mask or compensate for a severe gradient by stretching the image differentially or clipping the shadow pedestal in a levels tool. The golden rule requires the linear image to be completely flat and neutral, using MGC or native GraXpert, before you leave Tab 0. A poorly subtracted linear background will turn into an uncontrollable monster of noise and chromatic casts after hyperbolic stretching.
  2. Isolate Chrominance from Deconvolution: When configuring BlurXTerminator in Tab 0, always keep Luminance Only enabled. Mathematical deconvolution of chromatic channels usually provides no real gain in resolution, because of color undersampling in Bayer matrices or seeing dispersion, and in exchange it injects severe high-frequency chrominance noise along stellar edges. Deconvolving luminance only guarantees sharp structures while maintaining smooth and continuous color transitions.
  3. Early Color-Noise Attenuation:[14] Take advantage of NoiseXTerminator's frequency-separation capability in Tab 2. Apply ruthless color-noise reduction through Denoise Color in the 0.80 to 1.00 range, combined with conservative luminance attenuation around 0.40 to 0.50. The human eye tolerates and even accepts a background with fine uniform luminance grain, but rejects low-frequency chromatic clouds and blotches immediately.
  4. Core Protection Through Sequential HDRMT: If you process extremely high-dynamic-range objects, for example the core of M31 or the Lagoon Nebula M8, apply HDR Multiscale Transform in Tab 2 using a two-pass strategy constrained by masks. Apply a first gentle pass with Number of layers = 7 to compress the outer crown, freeze the result in Memory 1, design a strict luminance mask that isolates only the burned central core, and apply a second HDRMT pass with Number of layers = 5 over that zone. This reveals internal filaments without flattening contrast in the rest of the galaxy.
  5. Differential Saturation in Narrowband: In the final combination tab, avoid raising the global saturation of an SHO compound linearly. Instead, use the Color Mask tool in Tab 2 to extract a highly selective mask aimed at the cyan/turquoise hue, pure OIII, typically centered at 180 degrees with a narrow 30-degree range and Sat min = 0.12. Store this mask in Mask Memory. Assign the combined layer to a slot, enable that local mask, and raise saturation. The faint oxygen structures will glow with vibrant electric blue without desaturating or contaminating the vast surrounding hydrogen clouds.
